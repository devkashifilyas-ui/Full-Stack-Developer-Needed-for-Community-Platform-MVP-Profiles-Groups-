Architecture overview
How Alliance v1 is structured and why — built to stay simple, designed to scale.
Stack

Frontend

Next.js / React — fast iteration, clean component model, easy hosting on Vercel or Netlify

Backend

Node.js API — stateless routes, simple auth middleware, easy to scale horizontally

Database

PostgreSQL — relational model for users, profiles, groups, memberships, requests, and tags

Data model

users

id, email, role, verified, created_at

player_profiles

id, user_id, display_name, avatar_url, games, platforms, region, playstyle, bio, roles_interests

groups

id, name, description, game, region, level, organizer_user_id, created_at

group_memberships

id, group_id, user_id, status (member/left), joined_at

join_requests

id, group_id, user_id, status (pending/approved/rejected), created_at, reviewed_at

group_tags

id, group_id, tag_type, tag_value

What is intentionally excluded from v1
MessagingSchedulingTournament bracketsStatsPaymentsVoice/chatAdvanced rolesAutomationsIntegrations
These are added only when usage proves the need. No speculative building.
Scaling path
→ Stateless API routes allow horizontal scaling from day one
→ Indexed relational tables handle read-heavy group browse and profile views
→ Asset storage separated from relational data (S3 or R2 for avatars)
→ NEST account unification — OAuth connection point ready in users table
→ Caching layer (Redis) added on group directory when traffic warrants
